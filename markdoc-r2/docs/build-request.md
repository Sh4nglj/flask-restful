

## **任务构造要求**

1. **硬性要求**：任务分类需要以下三种的一种  
   - **界面功能**：创建新的 UI 组件、页面或交互功能  
   - **运行时异常修复**：修复程序执行过程中的空指针、网络超时等运行时异常  
   - **业务逻辑**：实现新的业务处理流程、规则或算法  
2. **建议要求**：需要构造尽量复杂的任务，争取让 Trae CN 无法完成任务，触发 Model Breaking  
5. **内容难度介绍**  

- **medium**：  
  - 某个复杂系统的子功能模块的开发，各功能逻辑相互耦合，逻辑相对复杂。  
  - 模拟物理碰撞效果的代码实现（小球弹跳，旋转等）。  
  - 纯前端或纯后端的代码实现，不涉及前后端联通，常见的接口开发。

- **hard**：  
  - 算法题的优化实现，对时间复杂度和空间复杂度有严格要求。  
  - 涉及前后端的全栈项目0~1开发，有详细的业务逻辑要求，以及前端风格和布局的描述。  
  - 涉及大量第三方集成和AI功能的开发

| 维度                           | Medium 级     | Hard 级           |
| ------------------------------ | ------------- | ----------------- |
| 代码量                         | 50-300 行     | ≥300 行（多文件） |
| 技术栈广度                     | 单一框架/语言 | 跨技术栈（≥3 种） |
| 优化要求                       | 基础优化      | 极致性能/资源优化 |
| 人工开发耗时预估<br>（熟练者） | 1-4 小时      | ≥4 小时           |

### 示例  

```
优化个人账号页面的功能，具体需求如下：  
1. 在 /settings/account 页面中添加当前用户所在城市的表单填写功能模块，确保该模块与页面现有设计风格保持一致  
2. 城市选择组件必须接入通用的城市选择公共API接口，以获取最新、最准确的城市信息数据  
3. 城市选择范围限定为中国国内城市，无需包含国际城市选项  
4. 实现省/地区/县市三级联动选择功能，确保选择逻辑正确且交互流畅  
5. 确保用户选择的城市数据能够正确上传至后端服务器，并在页面重新加载或用户再次访问时能够准确回显已保存的城市信息
```
